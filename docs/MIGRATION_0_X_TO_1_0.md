# 从 0.x 迁移到 1.0

言台 1.0.0 不删除或重命名 0.9.0 的公开方法，也不提升 ABI 或协议主版本。0.9.0 应用只需
更新依赖与锁文件；从更早 0.x 升级的应用还应检查中间版本已经引入的容量、生命周期和
能力协商规则。

## 一、升级依赖

宿主必须是言序 `>=1.1.7, <2.0.0`，项目清单使用格式 2，并保留真实需要的权限。Git
依赖可以固定 1.0 标签：

```toml
[依赖]
言台 = { 包 = "yanxu-platform", git = "https://github.com/yanxulang/yanxu-platform.git", 修订 = "v1.0.0", 版 = "^1.0" }

[权限]
图形界面 = true
原生扩展 = true
剪贴板 = true       # 只在使用剪贴板时授予
文件对话框 = true   # 只在使用文件对话框时授予
```

随后显式更新并检查：

```sh
yanbao 更
yanbao 查
yanxu 查 src/主.yx
```

提交清单与重新生成的`言序.lock`，不要手工修改锁文件中的目标、动态库路径、SHA-256、
大小或包内容摘要，也不要把旧版动态库复制到 1.0 清单下。

## 二、启动时协商

创建应用前的初始化阶段先读取`协议查询（）`和`能力查询（）`。1.0.0 返回 ABI 2、平台 1.7、事件
1.3、无障碍 1.0、绘制 1.1。消费者应：

1. 精确拒绝未知 ABI 或协议主版本；
2. 对自己要使用的新增能力检查相应次版本或能力字段；
3. 忽略未知可选字段与事件，并继续处理同一批次中的其余事件；
4. 按能力返回的容量构造图片、文字、帧和无障碍树，不复制文档中的数值作为跨后端假设。

机器契约见 [`api/native-abi-v2.json`](../api/native-abi-v2.json)、
[`api/protocol-contract-v1.json`](../api/protocol-contract-v1.json)与
[`api/error-codes-v1.json`](../api/error-codes-v1.json)。

## 三、按起始版本检查

| 起始版本 | 升级到 1.0 时需要确认 |
| --- | --- |
| 0.9 | 无源码迁移；更新依赖和锁文件，在应用声明支持的各目标重新执行平台测试 |
| 0.8 | 0.9 只增加稳定性、性能与供应链门禁；运行时协议不变 |
| 0.7 | 应用资源现在有默认硬配额；只能在首个子资源或首次运行前下调；退出后不能再次运行 |
| 0.6 | 六目标已启用 UIA、NSAccessibility 或 AT-SPI；语义树会真实进入系统辅助技术 |
| 0.5 | 无障碍协议增加有界语义树、焦点/动作请求和 1.3 事件；需要为控件提供稳定节点编号 |
| 0.4 | 可改用帧反馈链驱动动画；被替换帧不会产生呈现事件，不能持续等待旧帧 |
| 0.3 | 剪贴板图片固定为 RGBA8；文字 16 MiB、图片单边 16384 和内容 256 MiB 都会严格拒绝超限 |
| 0.2 | 每窗口只保留一个待呈现帧；事件、资源和帧统计可用于检测生产速度与背压 |
| 0.1 | 状态锁故障会恢复并计数；捕获的后端恐慌不再让后续清理重复失败 |

这些版本都保持 ABI v2 和既有方法签名。表中“需要确认”主要是运行行为、能力使用和错误
处理，不表示必须启用所有新增功能。

## 四、必须复核的行为

### 资源与关闭

- 应用、窗口、计时器、图片和字体仍是带类型资源；关闭父资源会先关闭子资源。
- `关闭（）`和`退出（）`可以重复调用，但已退出应用不能重新进入事件循环，会返回
  `PLATFORM_APPLICATION_EXITED`。
- 默认应用配额等于 0.7 及更早版本的既有安全边界。若应用此前接近边界，应读取
  `应用.资源配额（）`和调试快照，而不是捕获错误后无限重试。

### 事件与动画

- 离散事件队列上限为 4096；满时返回`PLATFORM_QUEUE_FULL`并停止循环，不静默丢事件。
- 指针移动、窗口移动/缩放和重绘只合并相邻最新值，滚轮相邻增量累加；离散事件保持顺序。
- 动画优先使用`提交帧反馈`与匹配的`帧呈现`。回执显示替换时，应立即丢弃被替换帧的等待
  状态。

### 无障碍

- 节点编号必须为窗口当前树内唯一的正整数；上层控件身份不变时保持编号稳定。
- 密码框不能把值写回语义树。焦点与动作事件在处理前仍需核对窗口、树修订和节点身份。
- 不需要无障碍交互的应用也应允许系统桥读取基本窗口与控件语义，不能依赖 0.6 的纯模型
  行为来隐藏已经提交的树。

### 错误处理

1.0 固定 70 个错误码，但诊断消息不是兼容接口。把类似`[PLATFORM_*] 消息`的异常拆分时，
按结构化代码或稳定方括号代码匹配，不比较完整中文消息；对未来 1.x 新增的未知代码保留
通用失败路径。

## 五、验收与回滚

升级至少执行：

- 当前目标的类型检查、字节码执行和真实窗口自动退出；
- 窗口创建/关闭、最后窗口退出、事件冲刷与队列压力；
- 实际使用的剪贴板、文件对话框、IME、字体和图片路径；
- 帧替换/呈现反馈，以及语义树更新和辅助技术动作；
- 安装产物中动态库目标、大小与摘要和锁文件完全一致。

需要回滚时，恢复升级前一起保存的清单和锁文件，再运行`yanbao 装`复核原锁；不要只替换
动态库。若 1.0 已写入应用自己的持久配置，应先确认这些配置仍只含普通兼容字段；言台的
诊断计数、资源句柄、事件序号、树修订和帧序号都不是跨进程持久化格式。
