# 兼容性政策

## 版本基线

| 项目 | 0.9.0 基线 |
| --- | --- |
| 言序 | `>=1.1.7`，已用 1.1.7 构建与执行 |
| 言包 | 不要求升级；包清单格式 2 |
| 原生 ABI | v2 |
| 平台协议 | 1.7 |
| 事件协议 | 1.3 |
| 无障碍协议 | 1.0 |
| 绘制协议 | 1.1 |
| Rust | 1.92，edition 2024 |

现有 `yanxu-gui`（言窗）不依赖言台，0.9.0 没有修改它的源码、清单或公开 API。言窗和
`yanxu-ui`（言界）可以并存；包名、模块名和原生资源类型均不冲突。

## 语义化版本

仓库版本遵循语义化版本。0.x 阶段仍通过协议主版本保护二进制数据：

- 修订版修复实现与文档，不改变已发布协议字段含义；
- 次版本可以增加包装方法、能力字段、事件或绘制操作码；
- 删除/重命名公开 API，改变单位、资源所有权或既有二进制布局必须先提升相应协议主
  版本，并在包版本中明确迁移窗口。

## 协议协商

应用启动时调用`协议查询`。ABI 必须等于 2，平台/事件/绘制主版本必须是消费者支持的
值。次版本按以下方式兼容：

- 典格式增加字段时，旧消费者忽略未知字段；
- 事件增加名称时，旧消费者忽略未知事件并继续处理批次其余部分；
- 绘制增加操作码时，旧后端按命令长度跳过；
- 新消费者不得假设旧生产者一定提供较新次版本字段。

平台协议 1.1 只新增`状态故障恢复`能力与`状态恢复次数`诊断字段；1.0 消费者按上述未知
字段规则保持兼容，既有字段的类型与含义不变。

平台协议 1.2 增加`运行可观测性`、`待呈现帧上限`能力以及事件队列、资源和帧的诊断典；
窗口查询增加`待呈现帧`与`帧字节`。这些都是可忽略字段，1.0/1.1 消费者行为不变。

平台协议 1.3 增加`剪贴板文字`、`剪贴板图片`、三项容量上限与`剪贴板图片格式`能力；
同时新增 RGBA8 图片读写方法。旧消费者仍可继续使用文字接口，新消费者必须先查询能力，
不能假定 1.0–1.2 后端支持图片或相同上限。

平台协议 1.4 增加帧提交回执、窗口帧身份、呈现反馈与动画驱动能力；事件协议 1.2 只新增
离散的`帧呈现`事件及其单调时间字段。既有提交方法仍返回命令数；旧消费者忽略未知事件
即可保持行为不变。

平台协议 1.5 增加无障碍协议版本、语义树更新/查询、硬上限和角色/状态/动作能力集合；
事件协议 1.3 增加经过当前树修订验证的焦点与动作请求。`原生无障碍桥`单独为假，因此
消费者可以先接入跨平台语义模型，但不能把 1.5 本身当成系统 UIA、NSAccessibility 或
AT-SPI 已可用的证明。

平台协议 1.6 把同一语义树接入 UIA、NSAccessibility 与 AT-SPI，新增值为真的
`原生无障碍桥`、平台后端名称和原生桥诊断字段。无障碍协议仍为 1.0、事件协议仍为 1.3；
既有树、动作参数和事件含义没有改变。1.5 消费者可忽略新字段，1.6 消费者必须检查能力，
不能只按平台次版本推断某个非桌面目标也有系统桥。

平台协议 1.7 追加配额查询/配置操作、应用资源硬上限、拒绝统计和生命周期能力。默认
配额等于 1.6 的既有硬边界，因此旧应用不配置时容量语义不变。新增调试字段可忽略；新
消费者若要下调配额，必须检查`应用资源配额可下调`并在首个子资源或首次运行前完成。
应用退出从可重复布尔标记收敛为单向状态机；既有`退出（）`仍无返回值且重复调用成功。

0.9.0 不增加操作、字段、事件或绘制命令，平台协议保持 1.7。它只增加畸形语料、浸泡、
Release 性能和零例外供应链门禁，因此 0.8.0 消费者无需修改代码；API 快照和 ABI v2
导出保持相同。

损坏数据、主版本不同或必需字段类型错误不是兼容情况，必须返回稳定错误而不是猜测。

## 六目标

正式包包含以下精确目标：

```text
x86_64-pc-windows-msvc
aarch64-pc-windows-msvc
x86_64-apple-darwin
aarch64-apple-darwin
x86_64-unknown-linux-gnu
aarch64-unknown-linux-gnu
```

上层言序代码和包入口完全相同；`言序.toml` 的原生索引按 OS/架构选择制品并验证 SHA-256
与字节大小。Linux 同一制品支持 Wayland 和 X11 回退。

## 数据与配置

平台公开值只使用言序基础数据类型，因此可以无损放入言据内存模型。配置持久化采用言据
时，加载后传给 API 的仍是普通典；JSON 入口先通过言据兼容转换。未知配置键默认保留或
忽略，不能影响已知键语义；资源配额是安全边界使用的封闭配置，未知键明确返回
`PLATFORM_QUOTA_CONFIG`。具体决策见
[ADR-004-yanju-json-compatibility.md](ADR-004-yanju-json-compatibility.md)。

## 已知限制

- 0.9.0 是桌面库，不支持移动端或 Web。
- CPU 绘制算法跨目标一致，但系统字体文件不同会改变文字像素；像素金图需携带同一字体。
- 文件对话框、剪贴板、真实 IME 候选窗和桌面主题依赖用户会话，纯无头 CI 只能验证可
  确定部分。
- 任意旋转文字、隔离离屏图层和 GPU 后端不在当前承诺内；密码文字选区不通过系统桥暴露。
- Linux 需要可用的 Wayland 或 X11 会话；CI 用 Xvfb 验证 X11 路径。
- UIA、NSAccessibility 与 AT-SPI 的最终播报和快捷键行为依赖用户启用的辅助技术、系统
  语言和桌面会话；无辅助技术的 CI 只证明适配器、树同步和窗口生命周期可运行。

## 弃用

新增替代接口后，旧接口至少在同一 0.x 次版本线保留并在文档与编译诊断中标为弃用。
协议字段不通过静默复用完成迁移。正式删除必须提供 CHANGELOG、迁移文档和可自动检查的
版本门禁。
