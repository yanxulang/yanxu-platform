# 兼容性政策

## 版本基线

| 边界 | 1.0.0 基线 | 机器门禁 |
| --- | --- | --- |
| 言序宿主 | `>=1.1.7, <2.0.0`；必须提供 ABI v2 | 六目标均用 1.1.7 构建、装包和执行 |
| 言包 | 包清单格式 2 | `言序.toml`与锁文件集成测试 |
| 言序公开 API | `api-v1.json` | 由言序 1.1.7 重新生成并逐字节比较 |
| 原生 ABI | v2，64 位小端 C 布局 | `native-abi-v2.json`与真实描述符比较 |
| 平台协议 | 1.7 | 查询字段、能力类型和资源硬上限快照 |
| 事件协议 | 1.3 | 批次、39 个事件名称和合并方式快照 |
| 无障碍协议 | 1.0 | 角色、状态、动作和硬上限快照 |
| 绘制协议 | 1.1 | 头布局、17 个操作码、路径码和上限快照 |
| 错误模型 | 70 个稳定代码 | 运行时源码扫描与错误码目录比较 |
| Rust 源码 | 1.92，edition 2024 | 格式、全目标测试与警告即错误 Clippy |

三份 1.0 机器契约见[原生 ABI 冻结](NATIVE_ABI_V2.md)、
[协议冻结](PROTOCOL_CONTRACT_1_0.md)和[稳定错误码](ERROR_CODES_1_0.md)。

现有 `yanxu-gui`（言窗）不依赖言台，1.0.0 不修改它的源码、清单或公开 API。言窗和
`yanxu-ui`（言界）可以并存；包名、模块名和原生资源类型均不冲突。

## 语义化版本

从 1.0.0 起，包版本和各协议版本分别表达不同兼容边界：

- `1.x.y`修订版修复实现、安全、性能和文档，不增加必需配置，不改变已发布成功调用的
  结果类型、单位、所有权或错误码含义；
- `1.x.0`次版本可以向后兼容地增加包装方法、可选能力字段、事件、无障碍词汇、绘制
  操作码或错误码，同时提升受影响协议的次版本；
- 删除或重命名公开 API、收紧普通有效输入、改变既有字段类型/单位、资源所有权、线程
  规则、C 布局或操作编号，只能进入 2.0 或新的协议/ABI 主版本；
- 为修复高危漏洞而必须拒绝此前不安全的输入时，可以在修订版收紧校验，但必须同时发布
  安全公告、稳定错误码和受影响输入的迁移方法。

1.x 中新增内容不能成为旧调用的隐含必需条件。消费者必须忽略未知可选字段和事件；生产
者必须查询能力，不能仅凭操作系统或包次版本推断后端能力。

包清单接受言序 1.1.7 起的 1.x 宿主，并显式拒绝尚未验收的 2.x。1.1.7 是每次发布的
最低集成门禁；更高 1.x 必须继续提供 ABI v2 和格式 2 包语义。未来言序主版本只有在独立
兼容 CI 通过并发布新的言台修订后才会进入支持范围。

## 协议协商

应用启动时调用`协议查询`。ABI 必须等于 2，平台/事件/绘制主版本必须是消费者支持的
值。次版本按以下方式兼容：

- 典格式增加字段时，旧消费者忽略未知字段；
- 事件增加名称时，旧消费者忽略未知事件并继续处理批次其余部分；
- 绘制增加操作码时，旧后端按命令长度跳过；
- 新消费者不得假设旧生产者一定提供较新次版本字段。

平台协议 1.1 只新增`状态故障恢复`能力与`状态恢复次数`诊断字段；1.0 消费者按上述未知
字段规则保持兼容，既有字段的类型与含义不变。

平台协议 1.2 增加`运行可观测性`、`待呈现帧上限`能力以及事件队列、资源和帧的诊断典；
窗口查询增加`待呈现帧`与`帧字节`。这些都是可忽略字段，1.0/1.1 消费者行为不变。

平台协议 1.3 增加`剪贴板文字`、`剪贴板图片`、三项容量上限与`剪贴板图片格式`能力；
同时新增 RGBA8 图片读写方法。旧消费者仍可继续使用文字接口，新消费者必须先查询能力，
不能假定 1.0–1.2 后端支持图片或相同上限。

平台协议 1.4 增加帧提交回执、窗口帧身份、呈现反馈与动画驱动能力；事件协议 1.2 只新增
离散的`帧呈现`事件及其单调时间字段。既有提交方法仍返回命令数；旧消费者忽略未知事件
即可保持行为不变。

平台协议 1.5 增加无障碍协议版本、语义树更新/查询、硬上限和角色/状态/动作能力集合；
事件协议 1.3 增加经过当前树修订验证的焦点与动作请求。`原生无障碍桥`单独为假，因此
消费者可以先接入跨平台语义模型，但不能把 1.5 本身当成系统 UIA、NSAccessibility 或
AT-SPI 已可用的证明。

平台协议 1.6 把同一语义树接入 UIA、NSAccessibility 与 AT-SPI，新增值为真的
`原生无障碍桥`、平台后端名称和原生桥诊断字段。无障碍协议仍为 1.0、事件协议仍为 1.3；
既有树、动作参数和事件含义没有改变。1.5 消费者可忽略新字段，1.6 消费者必须检查能力，
不能只按平台次版本推断某个非桌面目标也有系统桥。

平台协议 1.7 追加配额查询/配置操作、应用资源硬上限、拒绝统计和生命周期能力。默认
配额等于 1.6 的既有硬边界，因此旧应用不配置时容量语义不变。新增调试字段可忽略；新
消费者若要下调配额，必须检查`应用资源配额可下调`并在首个子资源或首次运行前完成。
应用退出从可重复布尔标记收敛为单向状态机；既有`退出（）`仍无返回值且重复调用成功。

0.9.0 不增加操作、字段、事件或绘制命令，平台协议保持 1.7。它只增加畸形语料、浸泡、
Release 性能和零例外供应链门禁，因此 0.8.0 消费者无需修改代码；API 快照和 ABI v2
导出保持相同。

1.0.0 不增加运行时操作或协议字段，继续使用上述 1.7/1.3/1.0/1.1 组合。它把公开 API、
ABI、协议词汇、操作码、上限和 70 个错误码提升为 1.x 机器兼容门禁；0.9.0 消费者无需
修改源码即可升级。0.x 使用者应按[迁移指南](MIGRATION_0_X_TO_1_0.md)逐项确认各次版本
首次引入的能力和生命周期约束。

损坏数据、主版本不同或必需字段类型错误不是兼容情况，必须返回稳定错误而不是猜测。

## 六目标

| 目标 | CI 资格环境 | 窗口 | 无障碍 |
| --- | --- | --- | --- |
| `x86_64-pc-windows-msvc` | `windows-2025` | Windows | UIA |
| `aarch64-pc-windows-msvc` | `windows-11-arm` | Windows | UIA |
| `x86_64-apple-darwin` | `macos-15-intel` | AppKit | NSAccessibility |
| `aarch64-apple-darwin` | `macos-15` | AppKit | NSAccessibility |
| `x86_64-unknown-linux-gnu` | `ubuntu-24.04` | Wayland/X11 | AT-SPI |
| `aarch64-unknown-linux-gnu` | `ubuntu-24.04-arm` | Wayland/X11 | AT-SPI |

上层言序代码和包入口完全相同；`言序.toml` 的原生索引按 OS/架构选择制品并验证 SHA-256
与字节大小。六项目标都是正式支持范围并运行真实窗口自动退出冒烟；表中资格环境是每次
发布实际验证的系统，不据此虚构未运行过的旧系统最低版本。Linux 同一制品支持 Wayland
和 X11 回退。

## 数据与配置

平台公开值只使用言序基础数据类型，因此可以无损放入言据内存模型。配置持久化采用言据
时，加载后传给 API 的仍是普通典；JSON 入口先通过言据兼容转换。未知配置键默认保留或
忽略，不能影响已知键语义；资源配额是安全边界使用的封闭配置，未知键明确返回
`PLATFORM_QUOTA_CONFIG`。具体决策见
[ADR-004-yanju-json-compatibility.md](ADR-004-yanju-json-compatibility.md)。

## 已知限制

- 1.0.0 是桌面库，不支持移动端或 Web。
- CPU 绘制算法跨目标一致，但系统字体文件不同会改变文字像素；像素金图需携带同一字体。
- 文件对话框、剪贴板、真实 IME 候选窗和桌面主题依赖用户会话，纯无头 CI 只能验证可
  确定部分。
- 任意旋转文字、隔离离屏图层和 GPU 后端不在当前承诺内；密码文字选区不通过系统桥暴露。
- Linux 需要可用的 Wayland 或 X11 会话；CI 用 Xvfb 验证 X11 路径。
- UIA、NSAccessibility 与 AT-SPI 的最终播报和快捷键行为依赖用户启用的辅助技术、系统
  语言和桌面会话；无辅助技术的 CI 只证明适配器、树同步和窗口生命周期可运行。

## 支持周期

- 1.x 是稳定兼容线；使用者应升级到所用次版本的最新修订版，旧修订版不单独回移普通
  缺陷修复。
- 最新 1.x 次版本接收正确性、安全和关键平台兼容修复；前一个次版本在后继次版本发布后
  至少 90 天继续接收高危与严重安全修复。
- 2.0 发布后，最后一个 1.x 次版本至少 12 个月继续接收高危与严重安全修复；结束日期会
  在 `SECURITY.md` 和 Release Notes 中提前公布。
- 0.x 不享受上述长期窗口；其已经公布的临时安全修复截止日期继续以`SECURITY.md`为准。

支持周期不扩大平台范围。未列入六目标的系统、浏览器、移动端、GPU 后端和第三方修改版
可以复用源码，但不作为官方 1.0 二进制兼容承诺。

## 弃用

1.x 新增替代接口后，旧接口在整个 1.x 中继续可用，并在 API 文档、CHANGELOG 和可行的
编译诊断中标为弃用。计划在 2.0 删除的接口至少提前一个 1.x 次版本给出替代方案与迁移
示例。协议字段、事件名、操作码、资源类型和错误码不通过静默复用完成迁移；正式删除还
必须提供迁移文档、兼容语料和新的主版本门禁。
