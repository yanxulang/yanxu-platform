# Release 性能预算

言台把性能回归门禁限定在固定的 Linux x86-64 CI 运行器上，并始终使用 Cargo
`release`配置。六个正式目标仍全部执行格式、测试、Clippy、构建和真实窗口冒烟，但不在
六种不同硬件上用一次短计时决定性能是否回归。

## 预算

| 场景 | 计量单位 | 中位数上限 |
| --- | --- | ---: |
| 绘制协议解码（每帧 256 条命令） | 单条命令 | 200 ns |
| 事件合并并冲刷（每批 256 个事件） | 单个输入事件 | 1,000 ns |
| 应用资源创建、运行、退出与关闭 | 单个资源 | 2,000 ns |
| 65 条命令的 256×256 CPU 栅格帧 | 单帧 | 750,000 ns |

这些上限用于发现数量级或持续性的回归，不是对任意用户机器延迟的承诺。真实窗口呈现还受
操作系统合成器、显示刷新率和驱动影响，不由这个进程内 CPU 套件代替。

## 稳定测量方法

运行器对每个场景先预热并自动增加迭代次数，直到校准样本至少持续 25 ms，再把正式样本
扩展到约 200 ms。随后丢弃两轮预热，记录九轮正式样本，以单位耗时中位数判定预算，并
要求中位绝对偏差（MAD）不超过中位数的 20%。因此单次调度抖动不会直接决定结果，噪声
过大的测量也不会被当作有效通过。

运行器会拒绝 debug 构建。报告采用版本化 JSON，记录包版本、系统、架构、迭代数、各项
中位数、MAD、预算和最终判定。CI 保存原始报告，便于发布复核和后续趋势比较。

从多仓工作区根目录执行：

```sh
cargo run --manifest-path yanxu-platform/native/Cargo.toml \
  --release --locked --example performance_budget -- \
  --report /tmp/yanxu-platform-performance-budget.json
```

## 预算变更规则

- 业务实现不得通过减少命令数、事件数、资源种类、画布尺寸或样本数来绕过门禁。
- 收紧预算需要保存变更前后的同机报告；放宽预算必须说明已测得的正确性、内存或兼容性
  权衡，并作为独立提交评审。
- CI 运行器镜像或硬件类型变化时先建立新旧重叠基线，再调整阈值，不把跨机器绝对值直接
  解释为代码回归。
- 性能通过不能替代语义测试、浸泡测试、配额检查或六目标真实窗口验收。
