# 言台 0.8.0

言台 0.8.0 为每个应用增加可协商资源配额和单向关闭状态机，使资源耗尽、创建失败、重复
退出和事件循环错误都具有确定结果与可观测证据。默认配额保持 0.7.0 的既有容量，未主动
下调的应用无需迁移。

## 协议与 API

- 平台协议从 1.6 提升至 1.7；ABI 保持 v2，事件协议保持 1.3，无障碍协议保持 1.0，
  绘制协议保持 1.1。
- 原生操作表只在末尾追加`配额查询`和`配额配置`，既有操作编号不变。
- 言序应用新增`资源配额（）`和`配置资源配额（配置）`。省略字段沿用当前值；配置只能
  在首个子资源或首次运行前完成。
- 能力查询增加应用配额、硬上限、拒绝统计、生命周期状态、退出幂等、关闭资源归零等
  明示字段。

默认硬上限：

| 项目 | 每应用上限 |
| --- | ---: |
| 资源总数 | 4096 |
| 窗口 | 64 |
| 计时器 | 2048 |
| 图片 | 256 |
| 字体 | 64 |
| 图片持有字节 | 256 MiB |
| 字体持有字节 | 128 MiB |
| 全部窗口保留帧 | 128 MiB |
| 全部窗口无障碍节点 | 65536 |
| 全部窗口无障碍文字 | 16 MiB |

未知字段、负数、资源总数为 0 或超过硬上限返回`PLATFORM_QUOTA_CONFIG`；配置冻结返回
`PLATFORM_QUOTA_LOCKED`。运行期超限使用具体的`PLATFORM_QUOTA_RESOURCES`、
`PLATFORM_QUOTA_WINDOWS`、`PLATFORM_QUOTA_FRAME_BYTES`等稳定代码。

## 原子资源创建

- 事件批处理支持全有或全无提交。可见窗口的`窗口显示`和`需要重绘`不会只留下前半批。
- 初始事件入队失败会恢复资源编号、父子关系、配额冻结、使用量和创建/高水位指标。
- 字体数量或字节不足时不会先修改字体数据库；成功预检后才解析并创建字体资源。
- 图片先预检资源槽，读取尺寸后验证最终 RGBA8 字节，并把解码分配预算限制为应用剩余
  图片字节；容量不足不会先解码大图。

## 可控关闭

应用状态明确为`就绪`、`运行中`、`退出请求`、`已退出`和`已关闭`。首次运行永久冻结
配额；并发运行返回`PLATFORM_EVENT_LOOP_RUNNING`，已退出后重启返回
`PLATFORM_APPLICATION_EXITED`。重复退出成功但不重复改变状态或唤醒事件循环。

事件循环创建、表面创建、回调、队列和呈现错误都会通过守卫清除活动标记，进入`已退出`
并保留稳定错误；正常返回记录为空错误。关闭应用递归清理所有子资源并进入`已关闭`。

调试快照新增：

- 配额上限、使用量、冻结状态和按原因/资源类别划分的拒绝计数；
- 当前应用状态、事件循环是否活动与最后退出错误；
- 运行开始、退出请求、重复退出、正常/失败退出、应用关闭、回收资源和资源归零总数。

## 验证

- 97 项 Rust 测试覆盖所有配额类别、配置解析单计数、原子事件、创建回滚、字体/图片预检、
  生命周期单向迁移、失败收敛和关闭指标。
- 128 轮混合关闭压力测试累计创建并关闭 1920 个窗口/计时器/图片/字体及应用资源；每轮
  都证明持有字节、帧、无障碍树和资源实时量归零，最终创建与关闭总数逐项相等。
- 言序 1.1.7 集成覆盖高层配额 API、稳定错误、能力字段和调试快照。
- 六个正式桌面目标执行格式、测试、Clippy、依赖策略、Release 构建、ABI v2 导出、言序
  集成和真实窗口自动退出门禁。

## 安装

Git 依赖可固定本标签：

```toml
[依赖]
言台 = { 包 = "yanxu-platform", git = "https://github.com/yanxulang/yanxu-platform.git", 修订 = "v0.8.0", 版 = "^0.8" }
```

最低言序版本仍为 1.1.7，应用仍需授予`图形界面`和`原生扩展`权限。

## 发布制品

Release 附带`yanxu-platform-0.8.0-six-targets.tar.gz`、独立 SHA-256 文件和完整
`yanxu-platform-0.8.0.toml`。归档与标签 Git 包包含相同的 Windows、macOS、Linux
x86-64/ARM64 动态库；清单逐项固定 SHA-256 与大小，来源记录指向生成它们的成功 CI。

本版本仍是桌面 CPU 二维后端，不承诺 GPU、移动端或 Web。完整配额、生命周期和错误
契约见[平台 API](PLATFORM_API.md)、[资源生命周期](RESOURCE_LIFETIME.md)与
[线程模型](THREADING_MODEL.md)。
