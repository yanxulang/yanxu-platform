# 安全政策

请通过 GitHub 仓库的 **Security → Report a vulnerability** 私下报告安全问题，不要在
公开 Issue 中附带利用细节。维护者会先确认影响范围，再协调修复、公告和版本发布。

`1.0.x` 接收正确性、安全和关键平台兼容修复；`0.9.x`、`0.8.x`、`0.7.x`、`0.6.x`、
`0.5.x`、`0.4.x`、`0.3.x`、`0.2.x` 与 `0.1.x` 的高危漏洞修复窗口持续到
2026-08-20。
原生扩展与宿主处于同一进程，
不是安全沙箱；应用仍必须显式授予`原生扩展`、`图形界面`以及剪贴板、文件对话框等
细分权限。所有 ABI 值、图片、字体、事件队列和绘制缓冲均设有大小或数量上限。

## 信任边界

- ABI v2 主机、应用自身言序代码和已安装原生动态库处于同一信任域。
- 绘制缓冲、无障碍树、图片、字体、配置和文件路径均视为不可信输入并在使用前校验。
- 言台不返回系统窗口句柄或原始指针；资源只能通过带类型 ABI 句柄访问。
- 不同事件循环/所有者线程的资源互用会失败，已关闭资源不会重新解释为其他类型。
- Rust panic 在 FFI 入口隔离并转换为`PLATFORM_BACKEND_PANIC`，不会跨 FFI 展开。
- 中毒的内部状态锁会在记录恢复次数后清除中毒标记，避免后续资源析构或事件循环二次崩溃。
- 原生辅助技术请求属于不可信输入；节点、当前树、状态、动作和参数在入队前重新校验。

## 硬上限

每应用最多 4096 个资源、64 个窗口、2048 个计时器、256 张图片和 64 个字体；图片、字体、
保留帧、无障碍节点和无障碍文字另有总量上限。绘制帧 16 MiB、单命令和文字字段 4 MiB、
事件队列 4096、单字体 64 MiB、解码图片最大边长 16384；图片解码分配还受应用剩余额度
约束。剪贴板文字为 16 MiB，RGBA8 图片单边 16384、总内容 256 MiB。
每个窗口只有一个待呈现帧槽。单窗口无障碍树最多 16384 节点，根为 0 时嵌套深度不超过
64，文字总量不超过 4 MiB，单节点最多 4096 个直接子节点与 16 个动作。浮点输入必须有限，
二进制长度和图片尺寸乘法使用检查运算，协议填充必须为零。超限输入返回稳定错误，不截断
后继续解释。

输入框原生文字运行从最多 65536 字节的已验证值生成。字符边界按 Unicode 字素计算，
异常长字素有界回退为标量；密码框不生成值或文字运行，敏感设置参数不进入诊断。

## 权限

创建应用检查`图形界面`；剪贴板读写和文件对话框分别检查同名细分权限。包清单还必须
启用 ABI v2 `原生扩展`，该权限由言序宿主在加载动态库时执行。权限失败不会降级成隐藏
的系统调用。

## 供应链

Cargo 依赖精确固定并锁定。CI 在六目标构建之外运行固定版本 `cargo-deny` 的公告、许可、
来源检查和 `cargo-audit --deny warnings`。GitHub Actions 用完整提交 SHA 固定。正式制品
记录 SHA-256 与大小，汇总包以固定排序、时间戳和所有者生成。

依赖审计规则、已修复公告和固定上游来源见
[`docs/THIRD_PARTY.md`](docs/THIRD_PARTY.md)。正式发布不得忽略已知漏洞、不健全公告、
停止维护公告、来源漂移或不兼容许可证。
